Findings**

- **High — GraphQL endpoint surface is not the same as Rust.**  
  Go serves playground on `GET /` and API on `POST /query` ([main.go](/Users/christophercaldwell/Code/projects/model_architecture/servers/go/cmd/graphql_server/main.go:41)), while Rust uses `GET /graphql` (playground) and `POST /graphql` (API) ([router.rs](/Users/christophercaldwell/Code/projects/model_architecture/servers/rust/crates/graphql_server/src/router/router.rs:13), [mod.rs](/Users/christophercaldwell/Code/projects/model_architecture/servers/rust/crates/graphql_server/src/router/graphql/mod.rs:15)).  
  Also, Go applies auth middleware to both routes ([main.go](/Users/christophercaldwell/Code/projects/model_architecture/servers/go/cmd/graphql_server/main.go:39)), while Rust keeps playground public and protects only POST ([router.rs](/Users/christophercaldwell/Code/projects/model_architecture/servers/rust/crates/graphql_server/src/router/router.rs:13)).

- **High — GraphQL error contract diverges from Rust.**  
  Rust maps domain failures to stable GraphQL errors with `extensions.code` (`NOT_FOUND`, `CONFLICT`, `INTERNAL_SERVER_ERROR`) ([mod.rs](/Users/christophercaldwell/Code/projects/model_architecture/servers/rust/crates/graphql_server/src/router/graphql/mod.rs:67)).  
  Go uses default gqlgen error behavior and wraps errors in resolver strings like `"get member: ..."`, with no equivalent `extensions.code` mapping ([resolver.go](/Users/christophercaldwell/Code/projects/model_architecture/servers/go/internal/transport/graphql/resolver.go:18), [schema.resolvers.go](/Users/christophercaldwell/Code/projects/model_architecture/servers/go/internal/transport/graphql/schema.resolvers.go:241)).

- **Medium — HTTP public surface is missing Rust’s docs endpoints.**  
  Rust exposes `/swagger-ui` and `/api-docs/openapi.json` ([router.rs](/Users/christophercaldwell/Code/projects/model_architecture/servers/rust/crates/http_server/src/router/router.rs:48)); Go HTTP router does not expose equivalent endpoints ([router.go](/Users/christophercaldwell/Code/projects/model_architecture/servers/go/internal/transport/http/router.go:17)).

- **Medium — Possible GraphQL scalar contract mismatch.**  
  Go schema defines `scalar Time` ([schema.graphql](/Users/christophercaldwell/Code/projects/model_architecture/servers/go/internal/transport/graphql/schema.graphql:1)); Rust GraphQL types use `chrono::DateTime<Utc>` with no custom scalar rename in source ([catalog/mod.rs](/Users/christophercaldwell/Code/projects/model_architecture/servers/rust/crates/graphql_server/src/router/graphql/catalog/mod.rs:17)). This likely means Rust exposes `DateTime` while Go exposes `Time`.

- **Low — HTTP error text casing differs from Rust.**  
  Go returns lowercase messages like `"something went wrong"` and `"member not found"` ([helpers.go](/Users/christophercaldwell/Code/projects/model_architecture/servers/go/internal/transport/http/httputil/helpers.go:31)); Rust returns `"Something went wrong"` / `"Member not found"` ([errors.rs](/Users/christophercaldwell/Code/projects/model_architecture/servers/rust/crates/http_server/src/router/errors.rs:46), [errors.rs](/Users/christophercaldwell/Code/projects/model_architecture/servers/rust/crates/http_server/src/router/errors.rs:67)).

**Architecture audit (CQRS / clean-onion)**  
I did **not** find business-rule leakage into Go transport/persistence layers; command/query split and domain decision placement are consistent with Rust’s intent. HTTP/GraphQL handlers are orchestration/mapping only, and decisions remain in domain + application command services.
